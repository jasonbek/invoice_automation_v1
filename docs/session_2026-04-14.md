## 1. Client-Facing Financial Disclosure (all booking types)

**Problem:** No consistent way to communicate deposit / supplier currency / CAD on client-facing messages. Financial summary was missing from invoiceRemarks entirely.

**Final rule (GLOBAL_RULES):** every Screen 1 `invoiceRemarks` (including hotels) prepends this block, for CAD and non-CAD invoices alike:

```
Payments are in [currency code]
The amount shown below is in CAD
Deposit: $[amount] [currency]
Total: $[amount] [currency]
Amount owing: $[amount] [currency]
```

- Deposit / Total / Amount owing use the **supplier's original invoice currency**, not CAD.
- If no deposit is shown, write `Deposit: $0 [currency]`.
- If amount owing is not explicit: compute `Total − Deposit`.
- Exchange rate stays in `agentRemarks`, never in the client block.
- Itinerary / booking detail never lives on Screen 1 — belongs on Screen 2 `clientFeedback` / `clientItinerary`.

## 2. Tour — Day-by-Day Itinerary on Screen 2

**Problem:** Screen 2 `clientFeedback` was returning category-grouped output (ACCOMMODATION / FLIGHTS blocks), not a chronological itinerary.

**Solution:** Rewrote the `clientFeedback` prompt to force a single chronological timeline synthesized from hotels + flights + transfers + activities. Header format:

```
Itinerary at a glance
----------------------

Day 1 — MM/DD/YY (City)
  Arrive [city] on flight [XX123] from [origin] at [time].
  Check in: [Hotel Name] ([room type]).
  [Meals / activities for that day].

...
```

- One entry per calendar day from `startDate` to `endDate`.
- "At leisure in [city]" fallback for days with no listed activity.
- Ends with a `Confirmation Numbers:` block.
- No pricing / deposits / commission anywhere on Screen 2.

## 3. Cruise — Day-by-Day Itinerary + Financial Placement

**Problem:** Cruise Screen 2 was leaking financial figures; itinerary had no structured day-by-day format.

**Solution (two parts):**
1. Added top-level FINANCIAL PLACEMENT RULE: all currency figures belong on Screen 1 only; Screen 2 contains zero currency amounts.
2. Rewrote `clientItinerary` to produce:

```
Itinerary at a glance
----------------------

Day 1 - MM/DD/YYYY - [Port / Embark]
Day 2 - MM/DD/YYYY - At Sea
Day 3 - MM/DD/YYYY - Nassau
...
Day 7 - MM/DD/YYYY - [Debark]
```

- No times (per user preference).
- Four-digit year.
- Optional `Inclusions:` line — non-financial only (packages, counts, never dollar amounts).

## 7. Hotel — Financial Info on Screen 1 + Multi-Hotel Split

**Problem A:** Hotel Screen 1 had no `invoiceRemarks` field at all, so the GLOBAL_RULES financial block was routed to Screen 2 `notesForClient` — which is for hotel contact info only.

**Fix A:**
- Added `invoiceRemarks` to Hotel Screen 1 schema.
- Updated GLOBAL_RULES: financial block goes to Screen 1 `invoiceRemarks` for every booking type (hotels included); no more hotels→notesForClient exception.
- Restricted Screen 2 `notesForClient` to hotel contact info + "Due at property" only — no deposit / totals / commission / savings.

**Problem B:** No formalized rule for invoices that contain multiple hotel stays under a single booking number (Expedia TAAP, BedsonLine). Behavior was working informally but wasn't written down.

**Fix B:** Added `MULTI_HOTEL_SPLIT_RULE`:
- Each distinct hotel stay = separate booking.
- Emit one `Screen 1 + Screen 2` pair per hotel, paired in order.
- Financials allocated per hotel, not from the invoice grand total.
- Shared invoice-level confirmation number reused unless a hotel-specific one exists.

Also added BedsonLine vendor rules + `bedsonline` ruleSet, and wired it into the routing agent.

## 8. Flight — Seat Block + Segment Date Continuity

**Problem A (seats):** The flight extractor was always emitting a "Seat Selections" block in `invoiceRemarks`, writing `Seat: N/A` lines when no seat data existed on the invoice.

**Fix A:** If no seats appear anywhere on the invoice, omit the seat block entirely — no header, no N/A lines. If some segments have seats and others don't, only list the segments that actually have assignments.

**Problem B (overnight dates):** On itineraries with overnight arrivals (e.g. YYZ 09/26 → LHR 09/27), the next segment's `startdate` sometimes reverted to 09/26 instead of 09/27.

**Fix B:** Added SEGMENT DATE CONTINUITY rule — each leg's `startdate` must be `>=` the prior leg's `enddate`. Connection times without explicit dates inherit the prior arrival date.
